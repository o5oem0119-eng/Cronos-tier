# Part 1. 서사 방향 (대본)

## 핵심 정체성
- 역사 정보 전달 채널이 **아니다**.
- 한 시대의 질서가 **어떻게 흔들리고, 무너지고, 대체되는지**를 보여주는 서사 채널이다.
- 스탯·티어는 채널의 중심이 아니다. **"메타가 어떻게 바뀌었는가"** 가 중심 서사다.

## 대본 기준 문서
- `directives/Chronos_Engine_Narrative_Formula_v4.md` 이 모든 대본의 최상위 지침이다.
- **몽골 제국 대본(`data/generated/mongol_empire/script.md`)이 앞으로의 방향을 보여주는 Gold Standard.**

## 대본 작성 원칙
- 내레이션에서 스탯·티어를 **절대 읊지 않는다.**
- 순수 서사 (이상 현상 → 기존 질서 설명 → 메타 붕괴 과정 → 새 질서 등장)로만 진행.
- 특정 타이밍에 `[VISUAL]` 태그를 삽입해 리모션이 보조 연출을 담당하게 한다.
- **`[VISUAL]` 태그는 씬(Scene) 단위로 작성한다. 한 씬의 실제 재생 시간은 해당 스크립트 구간의 녹음 파일 길이로 자동 결정된다.**
- **오디오 없는 테스트 시**: 씼당 기본값 10~15초를 플레이스홀더로 사용. `scene_density_fixer.py`로 자동 보간 가능.
- 썸 수량에 대한 최소 기준은 없다. **오디오 로딩 후 각 씬이 담당하는 다이얼로그 구간이 자연스럽게 씨 분할 기준이 된다.**

# Part 2. 비주얼 디렉션 (확정)

## 2-1. 배경 — 씬별 전환 전략 (확정)

> **파일 위치:** 배경 영상/이미지 파일들은 `assets/` 폴더에 저장.

배경은 하나로 고정하지 않는다. **장면(Phase)의 성격에 따라 아래 3가지 텍스처를 스위칭**한다.
배경 교체 자체가 시청자에게 "챕터가 바뀌었다"는 시각적 신호 역할을 한다.

> **소재 통일 원칙:** 전체 배경은 **구겨진 종이(Crumpled Paper)** 질감 한 가지 소재 안에서 명도만 다르게 사용한다.
> 소재를 통일함으로써 배경이 전환되어도 채널 전체가 하나의 톤앤매너 안에 있는 느낌을 유지한다.

| 배경 | 소재 | 사용 시점 |
|------|------|-----------|
| **1번 (다크 페이퍼)** | 어둡게 구겨진 검은 종이 질감 | **메인 배경.** 내레이션, 팩트 설명, 사료 분석 구간 |
| **2번 (라이트 페이퍼 A)** | 밝고 부드럽게 구겨진 흰 종이 질감 | **캐릭터 등장 씬.** 흰색 몸 + 검은 선화가 살아나는 배경 |
| **3번 (라이트 페이퍼 B)** | 밝고 강하게 구겨진 흰 종이 질감 | **챕터 전환·분위기 반전 구간.** 크로스페이드로 전환 |

- **형태:** 정지 이미지가 아닌 **루핑 영상(Video Loop)** 사용을 기본으로 한다.
- **핵심 이유 (2·3번 배경):** 캐릭터의 검은 선화가 어두운 배경에 묻히지 않도록, 캐릭터 등장 씬에는 밝은 페이퍼 배경을 사용한다. 별별역사도 동일한 밝은 구겨진 종이 배경을 사용하며 이미 시청자에게 검증된 방식이다.

## 2-2. 캐릭터 (차별화 핵심)

### 기본 구조
```
얼굴: 해당 국가/세력/인물을 대표하는 동물 (수인 형태)
몸:   사람 형태 + 시대적 의상과 악세사리
뱃지: 가슴 또는 어깨에 해당 국가의 국기/상징 마크를 작게 부착
```

### 차별화 포인트 (vs 경쟁 채널)
| 채널 | 캐릭터 형태 |
|------|------------|
| 별별역사 | 흰 몸 + 얼굴에 별 모양 + 간단한 선화 |
| 지식해적단 | 사람 몸 + 얼굴에 국기 (아주 가끔 사용) |
| **크로노스 티어** | **흰 몸 + 동물 머리 + 시대 의상/악세사리 + 국기 뱃지** |

### 동물 선정 원칙
- 현존 국가: 해당 국가를 대표하는 국가 동물 또는 상징 동물
- 멸망한 국가: 해당 문화권/지역을 연상시키는 동물
- 예시:
  - 몽골 → 늑대 (칭기스칸도 동일 캐릭터 사용)
  - 유럽 기사단 → 사자
  - 로마 → 독수리
  - 페르시아 → 사자/공작

### AI 이미지 생성 스타일 (확정: 낙서/Doodle체)

> **확정 스타일:** 정교한 일러스트가 아닌 **의도적으로 하찮은 낙서(Doodle)체**를 사용한다.
> 이유: "진지한 역사 분석 + 코믹한 캐릭터"의 긴장 구조가 채널 차별성을 만든다.
> 경쟁 채널(별별역사 등)과 명확히 구별되며 바이럴 가능성이 높다.

### 캐릭터 생성 3단계 파이프라인 (확정)

```
[0단계] 유니버설 레퍼런스 준비 (확정)
  위치: assets/characters/_template/character_reference.png (4번 낙서체 이미지)
  기존 보유 2인 이미지를 크롭(반 자르기)하여 동물별 단독 이미지로 분리 저장
  → assets/characters/_template/{animal}_ref.png

[1단계] img2img + 프롬프트로 새 캐릭터 변형
  레퍼런스 이미지 첨부 + 변경 항목 지정:
  - 동물 얼굴 종류 (늑대 → 독수리 등)
  - 해당 동물의 실제 체색 (털 색상, 깃털 색상 등 자연 색상으로 변경)
  - 시대 의상 (몽골 → 페르시아 등)
  - 국기·문양 뱃지
  또는: "1명만 나오도록" 프롬프트로 명시
  ※ 중요: 얼굴이 바뀌면 반드시 "해당 동물의 자연 체색에 맞게 색상도 변경" 명시

[2단계] 낙서 변환 프롬프트 적용
  "첨부한 이미지를 개발새발 세상에서 제일 하찮은 선으로 그려줘.
  배경은 흰색, 그림판에서 마우스로 그린 것 같은 픽셀 단위 그림.
  야 됐고 그냥 니맘대로 그려."

[완성] assets/characters/{nation}/doodle.png 저장
       배경 제거(누끼) → PNG 투명 배경
```

**핵심 장점:**
- 기존 이미지 크롭으로 레퍼런스 추가 제작 비용 0
- 몸 비율·포즈·반신 구도 자동 고정 → 전 캐릭터 일관성 유지
- 프롬프트만 바꿔 얼굴 + 의상 + 뱃지 교체

### 1차 제작 목표 (파일럿)
- **기존 보유 이미지(늑대/개 2인 컷)를 크롭하여 각 단독 레퍼런스 확정**
- 낙서 변환 결과물을 파일럿으로 검수
- 완성 후 이후 모든 캐릭터 제작의 기준 프로세스가 된다.

## 상시 등장 컴포넌트

### 1. `<WobblySpeechBubble />` — 물결 말풍선

**사용 위치 (2가지):**
1. **캐릭터 위:** 동물 캐릭터 옆에 붙어서 감정·반응을 코믹하게 표현
2. **역사 사료 위:** 실제 역사 자료(사진·문서) 위에 오버레이로 올라가서 당시 인물이나 세력의 반응을 재치있게 표현

**규격:**
- 1개 또는 2개 사용 가능 (대화 형식 가능, 2개가 필수는 아님)
- 외곽선이 물결치듯 꿈틀거리는 애니메이션 (Wiggle effect)
- 색상: 흰색 바탕, 두꺼운 검은 외곽선

---

### 2. `<HighlightText />` — 형광펜 긋기 모션
- 역사 사료나 텍스트의 핵심 부분에 형광펜이 그어지는 효과
- CSS width 0% → 100% 애니메이션으로 구현
- 시청자 시선 통제 + 분석가 느낌 극대화

---

### 3. `<Pictographic />` — 인포그래픽 픽토그램
- SVG 벡터 아이콘 라이브러리(Lucide, Phosphor Icons 등)를 코드로 내장
- 직접 다운로드한 PNG가 아니라 **코드로 렌더링**하여 채널 톤앤매너 강제 통일

**아이콘 색상:**
- **단일 색상 고정:** 흰색(`#FFFFFF`)
- 이유: 1번·3번 배경(어두운 질감) 위에서 단일 흰색이 가장 선명하게 보임

**애니메이션 규격:**
- `spring()` 모션만 사용 (자연스럽게 통통 튀며 등장)
- SVG Line Draw 효과 **없음**
- 카운트업 모션 **없음**

---

### 4. `<SourceBadge />` — 출처 표시 박스
- 사료(흑백 사진·문서·지도)가 화면에 등장할 때 **해당 사진 우측 하단에 부착**
- 항상 떠 있는 워터마크가 **아니다.** 사료가 나올 때만 붙는 **사료 전용 UI**
- **사료와 동시에 딱 나타남. 모션 일절 없음. 사료와 한 몸처럼 취급.**

**디자인 규격:**
- 배경: **검은색 반투명** (투명도 높여서 전체적으로 톤다운)
- 텍스트: 흰색, 작은 사이즈
- 예시: `출처: 위키미디어 커먼즈 / 국립중앙박물관`

### 6. `<StatCard />` — 분석 카드 팝업

**스타일 방향 (확정):**
> **공식 레퍼런스:** `assets/ui_references/statcard_reference.png` (2번 낙서 스타일 이미지)
> 캐릭터와 동일한 **낙서/손그림 스타일**로 제작한다.

- 카드 상단에 해당 **동물 캐릭터 또는 인물 아이콘** 포함
- ▲▼ 기호로 수치를 직관적으로 표현
- **네온 절대 금지**

# Part 4. `[VISUAL]` 태그 규칙 및 예시

## 핵심 원칙
- 씬(Scene) 단위로 작성한다. 한 씬 ≈ 약 10초.
- **한 장면에는 심플한 조합만 사용한다.**
- 사료 + 캐릭터 + 말풍선 + 픽토그램을 **한 장면에 동시에 구겨 넣지 않는다.**

## 가능한 조합 예시 (이 외에도 상황에 따라 자유롭게 가능)
```
조합 1: 사료 이미지 + 출처 + 말풍선
조합 2: 사료 이미지 + 출처 + 형광펜
조합 3: 캐릭터 + 말풍선 + 텍스트
조합 4: 픽토그램 + 텍스트
조합 5: 배틀씬 단독
조합 6: 스탯 카드 단독
```

```markdown
## 씬: 유럽 기사 비용 구조 (~10초)

(내레이션)
느리지만 단단한 방식. 당시 세계의 정석 빌드였습니다.
기사 한 명을 유지하는 데도 엄청난 비용이 들었습니다.

[VISUAL]
  배경: 1번 (다크 스톤)
  조합: 사료 이미지 + 출처 + 말풍선
  사료: 중세 유럽 기사 역사 이미지
  출처: 위키미디어 커먼즈
  말풍선: "유지비가 너무 비싸..."
[/VISUAL]
```

# Part 1. 철학 원칙

---

## 1. 크로노스식 역사 대본의 본질

크로노스식 대본은
역사 정보를 설명하는 콘텐츠가 아니다.

한 시대의 질서가
어떻게 흔들리고,
무너지고,
대체되는지를 보여주는 서사다.

핵심은 사건이 아니라 변화다.

누가 승리했는가보다,
왜 기존 세계가 더 이상 작동하지 않았는가가 중요하다.

---

## 2. 시작은 "설명"이 아니라 "이상 현상"

좋은 도입부는 배경 설명으로 시작하지 않는다.

먼저:
- 이해되지 않는 결과
- 모순적인 현상
- 기존 상식으로 설명 안 되는 상황

을 던진다.

그리고 시청자가

"왜 저게 가능하지?"

라는 상태가 되게 만든다.

도입부의 목적은 정보 전달이 아니라
의문 생성이다.

---

## 3. 기존 질서는 반드시 설득력 있어야 한다

기존 시스템은 멍청하면 안 된다.

당시 사람들이 왜 그것을 믿었는지,
왜 실제로 효과적이었는지를 먼저 보여줘야 한다.

그래야 이후 변화가 혁명처럼 느껴진다.

좋은 역사 서사는
과거를 비웃지 않는다.

오히려:

"그 시대 기준에서는 합리적이었다"

를 보여준다.

---

## 4. 핵심은 "개인"보다 "시스템"

크로노스식 대본은
영웅담이 아니다.

중요한 건:
- 어떤 구조가 작동했는가
- 왜 강했는가
- 왜 상대가 대응하지 못했는가
- 왜 결국 한계에 부딪혔는가

이다.

인물조차 시스템의 일부로 다룬다.

---

## 5. 모든 강점은 동시에 약점이 된다

좋은 역사 서사는
강점과 몰락 원인이 연결된다.

처음에는 효율이었던 것이,
시간이 지나면 버그가 된다.

강했던 이유가
나중에는 붕괴 원인이 되어야 한다.

---

## 6. 전개는 "적응 속도"의 차이를 보여준다

역사의 승패는 단순 전투력이 아니다.

누가 더 빨리:
- 변화에 적응했는가
- 새로운 방식을 받아들였는가
- 기존 규칙을 버렸는가

가 중요하다.

크로노스식 대본은
"강함"보다 "적응"을 추적한다.

---

## 7. 질문-반박 구조는 계속 세계관을 흔들어야 한다

시청자의 예상이 계속 무너져야 한다.

"곧 무너지겠지."
안 무너집니다.

"이건 통하지 않겠지."
오히려 더 강해집니다.

이 리듬은 단순 문체가 아니다.

기존 세계관이 붕괴되는 감각을
반복적으로 만드는 장치다.

---

## 8. 설명보다 장면이 먼저다

대본은 정보를 읽게 만들면 안 된다.

장면이 보여야 한다.

- 행동
- 반응
- 충돌
- 붕괴
- 추격
- 혼란

설명문보다
움직이는 장면으로 느껴져야 한다.

---

## 9. 게임 용어는 해석이 아니라 압축이다

게임 용어는 앞에 오면 안 된다.

먼저 현실을 보여준다.
그다음 구조를 설명한다.
마지막에 짧게 압축한다.

게임 용어는:
- 비유가 아니라
- 설명이 끝난 뒤의 한 줄 요약

역할만 해야 한다.

---

## 10. 완벽한 존재는 등장하면 안 된다

반드시:
- 실패
- 약점
- 한계
- 막히는 지점

이 있어야 한다.

그래야:
- 흡수
- 학습
- 진화
- 구조 변경

이 설득력을 가진다.

---

## 11. 역사적 아이러니는 핵심 장치다

깊이 있는 역사 서사는
반전된 결과를 남긴다.

- 파괴자가 연결자가 된다
- 질서를 무너뜨린 존재가 새 질서를 만든다
- 강점이 붕괴 원인이 된다

예상과 다른 결과가 나와야
대본에 여운이 생긴다.

---

## 12. 결말은 교훈이 아니라 질문이다

좋은 엔딩은 결론을 내리지 않는다.

오히려 현재에도 이어지는 질문을 남긴다.

- 강한 쪽이 살아남는가
- 적응한 쪽이 살아남는가
- 변화하지 못하는 순간 무너지는가

역사는 끝났지만,
질문은 현재까지 이어져야 한다.

# Part 2. 실행 규칙

> 철학은 "왜"를 다룬다.
> 실행 규칙은 "어떻게"를 다룬다.
> 아래는 실제 대본 작업 시 따르는 구체적 규칙이다.

---

## R1. 훅 구조 — 7단계

훅은 대본의 생명선이다.
반드시 아래 순서로 구성한다.

```
1. 조건의 모순    — 불가능해 보이는 조건을 나열한다
                    "성벽도, 도시도, 보급도 없었습니다."
                    "당시 기준으로는 절대 될 수 없는 조건이었죠."

2. 균열           — "그런데 이상하게도." 한 줄로 전환한다

3. 결과 나열      — 실제로 일어난 일을 압축해서 보여준다
                    중국. 페르시아. 러시아. 유럽.

4. 이름 공개      — 주인공을 마지막에 드러낸다
                    절대 처음에 밝히지 않는다

5. 호흡 포인트    — 이름 공개 직후 반드시 --- 삽입
                    나레이터와 편집자 모두를 위한 컷 포인트

6. 프레임 선언    — "단순한 정복 이야기가 아닙니다."
                    이 영상의 관점을 선언한다

7. 메타 브레이커  — 훅의 마지막 한 줄은 게임 용어로 압축한다
                    "인류 최초의 메타 브레이커 이야기입니다."
```

**훅 금지:**
- 주인공 이름으로 시작하지 않는다
- 업적 나열로 시작하지 않는다
- "오늘은 ~에 대해 알아보겠습니다" 식으로 시작하지 않는다

---

## R2. 월드빌딩 — 브릿지 문장 규칙

훅 다음, Phase 시작 전 구간이다.

**순서:**
1. 시대를 한 줄로 선언한다 — `"13세기."` 처럼 연도만
2. 당시의 정석 빌드를 구체적으로 보여준다
3. 그 빌드가 실제로 강했던 이유를 인정한다
4. 그 안에 숨겨진 구조적 약점을 드러낸다
5. 브릿지 문장으로 마무리한다

**브릿지 문장이란:**
월드빌딩의 마지막 문장.
주인공 세력의 등장을 예고하고 Phase로 넘어가는 연결점이다.

```
좋은 예:
"그런데 어느 날. 초원에서 전혀 다른 방식의 세력이 등장합니다."

나쁜 예 (브릿지 없이 바로 Phase 제목):
"--- ## Phase 1 ..."
```

---

## R3. Phase 제목 규칙

Phase 제목은 기술 설명이 아니라 극적 진술로 짓는다.

```
좋은 예: Phase 1 — 속도가 공격력이 되다
나쁜 예: Phase 1 — 몽골군의 기동성 분석
```

---

## R4. 나열은 반드시 한 줄씩

나열이 한 줄에 묶이면 나레이션 타이밍이 죽는다.

```
❌ 잘못된 예:
정보전에도 능했고, 조직 운영 효율도 높았으며, 흡수 속도 역시 빨랐습니다.

✅ 올바른 예:
정보전에도 능했고,
조직 운영 효율도 높았으며,
흡수 속도 역시 빨랐습니다.
```

한 줄씩 끊어야 편집자가 각 항목에 비주얼을 배치할 수 있다.

---

## R5. --- 삽입 기준

`---`는 무조건 많이 쓰는 게 아니다.
아래 세 가지 경우에만 삽입한다.

1. **감정 클라이맥스 직후** — 이름 공개, 반전 등 충격적인 한 마디 다음
2. **섹션 전환** — 월드빌딩 → Phase, Phase → 다음 Phase
3. **패치노트 전후** — 게임 연출 구간은 앞뒤로 분리

---

## R6. 패치노트 형식

시대의 규칙이 바뀌었음을 선언할 때 사용하는 게임 연출 구간이다.

**위치:** 전환점 직후, 다음 Phase 시작 전

```
화면에 패치노트가 뜹니다.

"Patch X.X.X — [시대를 반영한 패치명]"

- [변화 항목 1]
- [변화 항목 2]
- [변화 항목 3]

그리고 당시 [관련 세력]은
이 변화가 얼마나 위험한지 아직 몰랐습니다.
```

**주의:** 영상 전체에서 최대 1~2회만 사용한다. 남발하면 효과가 죽는다.

---

## R7. 기획 전 사고 프레임 — 9문

대본을 쓰기 전에 아래 질문에 먼저 답한다.

```
1. 그 시대가 믿고 있던 상식은 무엇인가?
2. 그 상식의 숨겨진 약점은 무엇인가?
3. 주인공은 그 약점을 어떻게 발견했는가?
4. 그는 어떤 새로운 방식으로 규칙을 바꿨는가?
5. 그 방식은 왜 압도적이었는가?
```
